# オントロジーを執筆工場に取り入れる — 調査＋適用可能性の見立て

作成日: 2026-07-06 / 狙い: ①世界設定の構造化 ②整合性の自動チェック ③AIへの文脈供給

---

## 0. 結論（3行）

- **相性は非常に良い。しかも新規導入ではなく「既存資産の formalize」で足りる。**
- taijiさんの `character_states.yaml`（knows / does_not_know / relationships / at_episode）は、実質「ID無しの手書きプロパティグラフ＋時間付き知識状態(epistemic state)」。オントロジーの手前まで来ている。
- 投資先は3つに集中する: **(a)矛盾の自動検出 (b)多段の関係クエリ (c)1話分の文脈切り出し**。OWL/RDFのフル導入・グラフDBは当面**不要**（むしろ創作の足枷）。

## 1. オントロジーとは（この工場に必要な範囲だけ）

「何が存在し、どう関係し、何が成り立たねばならないかを機械可読に書き下したもの」。構成要素5つ:

| 要素 | 物語での例 | 既存yamlでの現れ方 |
|---|---|---|
| Class（型） | Character / Location / Event / Item / Faction / Scene | ファイル種別・`type` |
| Instance（個体） | `ch_renji_old`, `王都` | yamlのキー |
| Property（値への関係） | `emotion`, `location`, `at_episode` | 既にある |
| Relation（個体への関係） | `KNOWS`, `MEMBER_OF`, `CAUSES` | `relationships`, `knows` |
| Constraint（制約＝矛盾検出） | 「死亡後キャラは登場不可」「知らない事実に言及不可」 | **ここが未実装＝伸びしろ** |

重要な区別:
- **taxonomy**（is-a階層1本）＜ **ontology**（関係＋制約＋推論）＜ 実装層が **knowledge graph**。
- 物語で主役になるのは「推論(axiom)」ではなく「**妥当性検査(constraint)＝矛盾検出**」。

技術選定（軽い順、推奨は最上段）:
- **★ YAML/JSON手書きプロパティグラフ + 薄いスキーマ + 自前バリデータ** … 個人プロジェクトの第一選択。DB不要。
- JSON-LD（`@id`/`@type`の発想だけ借用） … 将来の相互運用を残したい時。通常は過剰。
- Neo4j等のグラフDB … 矛盾検出・多段クエリが手に負えなくなってから移行。
- **RDF/OWLフル … 当面不要**（習得コスト高・開世界推論が即興創作を阻害・計算爆発）。

補足: RDFトリプルは「関係に属性を付ける」のが苦手（`KNOWS{since:3日目}` が書きにくい）。物語は関係に時刻・経緯を紐づけたい場面が多く、**プロパティグラフが構造的に自然**。

## 2. 既存資産の点検 — どこまでオントロジー的か

### すでにできていること（強い）

`fools/state/character_states.yaml` 抜粋:
```yaml
ch_renji_old:
  name: "佐山レンジ（老年期）"
  at_episode: "post-ep001"      # ← 時間軸(valid-time)
  location: "..."               # ← property
  emotion: "..."                # ← property
  knows: [ ... ]                # ← epistemic state（肯定）
  does_not_know: [ ... ]        # ← epistemic state（否定）★これが希少で強い
  relationships:                # ← relation（辺）
    ch_henshusha_wakai: "担当編集者として接する"
```
- **entity, property, relation, 時間軸, 知識状態が既に揃っている**。
- `does_not_know` を明示しているのは特に良い。「まだ知らないはずの事実に言及した」という矛盾検出の土台になる。

`foreshadowing.yaml`: `planted → hinted → strengthened → recalled` の状態遷移 + `intended_first_setup`/`intended_recall`（伏線の setup→payoff）。→ **未回収伏線の検出クエリがそのまま書ける形**。

`three_layer_status.yaml`: 事件ID（`01-02`）ごとに truth/perception/record の3層 drafted フラグ。→ 3層の埋め残し検出が可能。

### 足りない4点（伸びしろ＝オントロジー化で埋まる）

1. **IDの正規化**: `ch_renji_old` は独自表記。全個体に安定IDを振り、relationの宛先をIDで参照する（＝参照整合性チェックの前提）。
2. **relationの語彙統制**: `relationships` の値が自由文。`KNOWS`/`ALLY_OF`/`ENEMY_OF`/`MEMBER_OF` 等の**限定語彙**にすると機械処理可能に。自由文は `note` に退避。
3. **制約(constraint)が未実装**: 矛盾検出のルールがどこにもない。
4. **横断クエリ手段がない**: 「事実Xを知る全キャラ」「未回収の伏線」を human が目視で追っている。

## 3. 適用設計案（3用途 × 段階導入）

### 用途A: 世界設定の構造化

- `bible/` の下に **薄いスキーマ**を1枚置く。Class は最小限（Character / Location / Item / Faction / Event / Scene）。理論固有概念（役札、怪異型、三層…）は専用Classを増やさず **`type:` フィールド（統制語彙）で分ける**（GOLEMの定石＝クラス爆発回避）。
- 既存の world.md / characters.md を、この薄いスキーマに沿った **entity ファイル群**に整理（`entities/characters/*.yaml`）。中身は今のyamlをほぼ流用でOK。

### 用途B: 整合性の自動チェック（← 最大の実利）

evaluation-revolution-lab の criterion と直結する。小さな Python バリデータ（`tools/ontology_check.py`）で以下を出力:

| チェック | 対応する既存の評価基準 | クエリの考え方 |
|---|---|---|
| 参照整合性（宛先ID不在） | — | relation先IDがentityに存在するか |
| **epistemic矛盾**（知らない事実に言及） | knowledge-state-fairness / 破綻ゲートG1 | `KNOWS.valid_at > 発言ep` を検出 |
| 知識の単調性違反（既知を再び「初めて知る」） | drafter-preflight Gate | knows集合が ep 進行で非単調 |
| 未回収の伏線（payoff無きsetup） | 伏線の設置と回収 | status≠recalled のまま最終ep |
| タイムライン循環 | 因果・時間整合 | Event の PRECEDES が巡回 |
| 三層の埋め残し | （fools固有） | *_layer_drafted に false |

→ これは **ハーネスの Sensor**（事後観測）そのもの。draft後や packet freeze前に回す。**gatingせず warning から**始めるのが安全（創作の例外を殺さない）。

### 用途C: AIへの文脈供給（Writing Adapter の精密化）

- Writing Adapter が今「1話分に圧縮」しているのを、**entity起点の k-hop 近傍抽出**で機械化する。
- 例: ep014 を書く → 登場entity（レティシア, ミラ, 教室）を起点に1〜2hop（関係先＋そのeptime時点の知識状態＋関わる未回収伏線）を自動抽出 → `context_pack.md` に注入。
- LLMに全Bibleを渡さず「今この1話に効く部分グラフだけ」を渡せる＝context engineeringの精密化。「Lost in the Middle」対策にも効く。

## 4. 段階導入プラン（無理なく育てる）

**Competency Questions を先に3〜7個決める**（これがスコープの上限になり、膨張を止める）。例:
- 「事実Xを、ep_N時点で知っているキャラは誰か？」
- 「ep_N時点で未回収の伏線は？」
- 「キャラAと組織Fの関係の経緯は？」
- 「このシーンに登場する全entityと、その時点の状態は？」

| フェーズ | やること | 成果物 | コスト |
|---|---|---|---|
| P0 | ID正規化 + relation語彙を10個以内に統制 | `entities/` 命名規約 | 小 |
| P1 | 薄いスキーマ1枚 + 既存yamlを載せ替え | `schema.yaml` | 小〜中 |
| P2 | バリデータ（参照整合性＋epistemic矛盾＋未回収伏線の3本だけ） | `tools/ontology_check.py` | 中 |
| P3 | Writing Adapter に k-hop 近傍抽出を接続 | context_pack自動生成 | 中 |
| P4（必要なら） | 規模が破綻したらNeo4j移行 / SHACL導入 | — | 大（当面不要） |

## 5. やってはいけないこと（過剰設計の警告）

- **早すぎる形式化**: 世界観が固まる前に硬く型を作ると、その型自体が負債化する。「物語と共に育てる」。
- **OWL推論エンジンの常時稼働**: 計算爆発（最悪 N2ExpTime）。創作に厳密推論(DL)はほぼ不要。
- **強い排他制約を早期に張る**: `敵とdisjoint味方` `職業cardinality=1` は寝返り・転職で即破綻。制約は必須性(`minCount 1`)と参照整合性(`sh:class`相当)の2種で8割カバー。
- **アッパーオントロジー(BFO/DOLCE)接ぎ木・深いis-a階層**: 個人創作では過剰。
- **判断基準**: 「その形式化はCompetency Questionのどれかに答えるのに必要か？ NOなら足さない」。

## 6. 3レイヤ工学との接続（既存の把握マップと整合）

- オントロジー/KG = **context engineering の中核データ構造**（何をLLMに見せるかの源泉）。
- バリデータ = **harness の Sensor**（事後の整合性観測）。
- 「draft → ontology_check → 矛盾あれば修正」= **loop**（TAKT workflowの1 stepに組み込める）。

→ つまりオントロジーは4つ目の別レイヤではなく、**既存3レイヤを貫く「共通データ基盤」**として入る。これが一番自然な取り入れ方。
