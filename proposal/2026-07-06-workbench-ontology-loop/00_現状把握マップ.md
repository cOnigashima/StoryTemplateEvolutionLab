# カクヨム執筆プラットフォーム 統合試作 — 現状把握マップ

作成日: 2026-07-06 / 目的: 5つの資産を解剖し、「執筆工場（ワークフロー）」再構築の土台にする

---

## 0. 結論（1枚要約）

このフォルダには、同じ思想（**Story OS = 会話ログではなくファイルを正本にする物語制作OS**）を、
役割の違う 5 つのプロジェクトが別々に体現している。

- **設計を進化させる装置** … `StoryTemplateEvolution`（テンプレ／Bible／ディレクトリ規約の正本）
- **評価基準を発見する装置** … `evaluation-revolution-lab`（レビュー観点・評価基準の探索ラボ）
- **知識の図書館** … `know_how_explore`（小説技法・AI活用・評価ルーブリックの文献棚）
- **実走中の作品①（TAKT運用）** … `fools-with-cheating`（72話・自動化寄り）
- **実走中の作品②（Cowork運用）** … `villainess-coc-survival`（packet単位・人間判断寄り）

再構築で組みたい 3 レイヤは、既にこの中に**部品として散在**している。
まだ「1つの工場」に統合されていない、というのが現状。

| 組みたいもの | 対応する既存概念（工学用語での位置づけ） |
|---|---|
| コンテキストエンジニアリング | Bible/Design/State 三層 + Writing Adapter（1話分に圧縮） |
| ハーネスエンジニアリング | .claude/ の agents・rules・skills、Gate 0/A/C、DoR/DoD |
| ループエンジニアリング | TAKT の workflow YAML + loop_monitors、レビュー2ラウンド制 |

## 1. 各プロジェクトの正体

### 1-1. StoryTemplateEvolution（＝正本・Bible・ディレクトリ規約の中核）

「最強テンプレを完成させない」進化型の設計体。3層構成：`current/`（正本 v4）/ `proposal/`（提案履歴）/ `archive/`（凍結）。

核となる概念:

- **kernel（11項目）**: 作品1本の削れない芯。logline / promise / protagonist_vector / conflict / stakes / change_model / causality / information_design / emotional_arc / style_voice / unit_tree。
- **Layer / Facet**: kernelは薄く保ち、world・characters・system・timeline 等の 17 facet を厚くする。
- **Unit入れ子**: Manuscript → Part → Arc → **Packet** → Episode → Scene → Beat。
- **Storage Trinity（三層永続）**: `bible/`（frozen設計・Patch経由のみ）/ `design/`（揺れる設計）/ `state/`（append-onlyの動的事実）。
- **status 11値**: filled / tentative / deferred / needs_author_decision / contradiction … 空欄を作らず状態で管理。
- **2つのAdapter**: **Intake Adapter**（raw企画 → bible/design/stateへの反映"案"に変換、直接投入禁止）、**Writing Adapter**（確定Bibleから1話分だけに圧縮 → episode_pack 4ファイル）。
- **DoR/DoD ゲート**: DoR-A（Bible完全性）→ DoR-B（Writing Pack整備）→ DoR-C（Packet凍結）→ DoD-E（公開承認）。
- **agents 18 / skills 7 / rules 6**: drafter・critic・continuity・release 等の雛形。

思想: **実走した作品から抽象化してテンプレを進化させる（Pull型・Retro駆動・凍結禁止）**。

### 1-2. evaluation-revolution-lab（＝評価基準の探索ラボ）

既存ルーブリックに当てはめるのではなく、**現物本文と実戦ログから再利用可能な評価基準を掘り起こす**研究所。2車線＝`revolution`（発見）/ `evolution`（定着）。

- **Source優先順位**: 現物本文 > work-local learning > 既存review > know_how_explore > factory docs。
- **候補2種**: `criterion`（何を見るか）/ `method`（どう読むか）。
- 発見済みの criterion 例: theme-turn-embodiment / knowledge-state-fairness / image-before-verdict / ending-afterimage-strength / dialogue-purpose-density / scene-sequel-balance。
- 発見済みの method 例: axis-separated-review（読者体験⊥技術整合）/ multi-pass-draft-review / before-after-delta-review。
- **昇格ルール**: 3セッション以上再利用・2作品以上で有効・反例確認通過・rewrite指示に落とせる → docs へ昇格。

### 1-3. know_how_explore（＝ノウハウ図書館）

- **AI小説評価基準設計.md**: 25項目ルーブリック（100点満点・重み付け）。文体/構成/意味/読者反応の4層。破綻ゲート3項目（因果時間整合・話者識別・文法）。閾値: 80+公開候補 / 60-79改稿 / 59-作り直し。
- **小説レビュープロンプト.md**: 30項目レビュー + 対話特化 + マクロ構造。必須ゲート G1因果 / G2話者 / G3文法。
- **小説執筆ノウハウ総覧.md**: プロット/テンポ(Scene・Sequel)/視点/描写/対話/伏線/推敲の7領域。Want-Needのズレ＝葛藤の自走。テーマ4段階（問い→仮説→反証→結論）。
- **AI小説生成ガイド.md**: AIは「書き手」でなく「編集環境」。文脈4層（憲法/Story Bible/章要約/シーン台本）。「毎章、重要情報を先頭に戻す」。法務（著30の4・類似性+依拠性・KDP申告）。

### 1-4. fools-with-cheating（＝実走作品①・TAKT運用・72話）

- ダークファンタジー。bible/design/state + writing/episode_packs（ep001-072 × 4ファイル）+ `.takt/` + `.adapter/`。
- **TAKT統合**: workflows/ に YAML、facets/ で部品化、runs/ に実行ログ、`tools/takt_*.py` でダッシュボード。
- レビュー: typed review + reader_panel（8ペルソナ: commercial_editor / confused_reader / dropoff_reader / general_reader / literary_editor / sensitivity_editor / veteran_novelist / web_serial_editor）+ action_register.yaml で採否一元管理。
- 特徴: 規模中心・自動化優先・全72話を先にWriting Pack化。

### 1-5. villainess-coc-survival（＝実走作品②・Cowork運用・packet単位）

- クトゥルフ風悪役令嬢ブラックコメディ。`story/`（seed・promises・canon-patch）/ bible / arcs / **packets（8状態遷移）** / scenes / drafts / reviews / learning。
- **packet = 1〜4話の実装可能単位**。ディレクトリ8状態: exploring → scoped → **frozen** → drafting → review → approved → published → retro_done。frozen は2段（初回freeze → dry-run draft → 再freeze。ディレクトリは同じ `frozen/`）。
- **2段freeze**と**canon-patch protocol**で変更を全追跡。**ペルソナ2ラウンド制**（1周目=表層、2周目=次層、3周目はROI逓減で不投入）。
- HUB.md（現在地1枚）+ template.manifest.json（テンプレ版契約）。
- 特徴: 安定性中心・人間判断維持・部分再構成に強い。

## 2. 5つの関係（エコシステムとして）

```
        know_how_explore（文献棚）
                 │ 仮説の材料
                 ▼
   evaluation-revolution-lab（評価基準を発見）
                 │ 昇格
                 ▼
   StoryTemplateEvolution（テンプレ・Bible・規約の正本）
        │ 新作の高いbaseline          ▲ 実走からの学び(Retro)
        ▼                            │
   fools-with-cheating   ／   villainess-coc-survival（実走作品）
   （TAKTで自動化）          （Coworkで人間判断）
```

- **下から上へ**: 実走作品の learning → 評価ラボが基準化 → テンプレに昇格。
- **上から下へ**: テンプレの正本 → 新作の初期状態に反映。
- **横**: fools と villainess は同じ Story OS の別実装（自動化 vs 人間判断）。

## 3. 3つの工学レイヤ（調査結果の要点）

- **コンテキストエンジニアリング**: モデルが「見る情報」を最適キュレーション。→ 既存の Bible/Design/State 三層と Writing Adapter がこれ。
- **ハーネスエンジニアリング**: Agent = Model + Harness。モデル以外の全環境（scaffolding・ツール・制約・フィードバック）。Guides（事前予防: rules/skills/AGENTS.md）+ Sensors（事後観測: テスト/lint/レビュー）。→ 既存の .claude/ 一式・Gate・DoR/DoD がこれ。
- **ループエンジニアリング**: 行動→観測→判断→反復のサイクル自体を設計。「プロンプトを打つ人」から「プロンプトを出す機械を作る人」へ（2026年6月に命名確立）。→ TAKT の workflow が実装。

**TAKT**（nrslib製OSS）: AIエージェントのオーケストレーション。「AIの見張り番をやめる＝決定権をAIからworkflowに移す」。YAMLで step/persona/遷移rule/loop_monitors/COMPLETE・ABORT を宣言。**Faceted Prompting**（Persona / Policy / Instruction / Knowledge / Output Contract の5facet）でコンテキスト肥大を防ぐ。`npm install -g takt`、運用は `takt`（対話計画）と `takt run`（自律実行）。小説応用: 著者=coder / レビュアー・編集=reviewer / 設定資料=Knowledge / 文体規約=Policy / 章フォーマット=Output Contract / 堂々巡り検出=loop_monitors / 最終判断=人間。

## 4. 現状のギャップ（再構築で埋めるべき点）

1. **正本が2系統に分岐**: fools=TAKT自動化、villainess=Cowork人間判断。ディレクトリ構造も微妙に違う（packet概念の有無、freeze段数）。→ どちらを標準の背骨にするか未決。
2. **評価基準が未接続**: evaluation-lab の criterion/method、know_how の25項目ルーブリックが、実際のレビュー工程（.claude/skills や TAKT workflow）に自動で刺さっていない。
3. **テンプレ進化ループが手動**: StoryTemplateEvolution の「実走→抽象化→昇格」が、人間の判断待ちで、ループとして回っていない。
4. **作業場所・作業ルールの統一未了**: どこで書き、どこでレビューし、何を正本とするかの単一ルールが5プロジェクトに分散。
