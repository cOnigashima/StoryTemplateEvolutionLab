# Learning Capture（学びの自動記録）

セッション中に以下のいずれかが発生したら、`learning/` にログを残す。
セッション終了を待たず、**発生した時点で即座に書く**。

## キャプチャ対象

### 1. インシデント（設計・執筆の失敗と修正）

- 因果順の矛盾、canon 違反、設定の衝突を発見・修正した
- 公開済み canon との不整合に気づいた（例: 禁止したつもりの語が既出 canon に存在）
- 過剰な意味づけをして author に戻された（例: motif に深い意味を持たせすぎた）
- draft で packet 要件が脱落した

→ ファイル名: `learning/YYYY-MM-DD-{slug}.md`

### 2. 方法論の発見・改良

- 新しいレビュー手法が生まれた
- preflight の項目が増えた
- 執筆ワークフローの改善案が出た
- プロンプトの書き方で効果的なパターンを発見した

→ ファイル名: `learning/YYYY-MM-DD-{slug}.md`

### 3. author フィードバックの要点

- author が「これ違う」「こうして」と方向修正した
- 設計思想・文体・禁則について新しい指示が出た
- 「これは良い」と明示的に評価されたアプローチ

→ ファイル名: `learning/YYYY-MM-DD-{slug}.md`

### 4. 再利用可能なプロンプト・テンプレート

- レビュー用プロンプト
- draft 前のチェックリスト
- scene card の書き方パターン

→ ファイル名: `learning/YYYY-MM-DD-{slug}.md`

## ログのフォーマット

「何が起きたか」だけでなく、**なぜ起きたか・どうやるべきだったか・根本対応は何か** を必ず書く。未来の自分（別セッションの Claude）が読んで同じ失敗を避けられる粒度にする。

```markdown
# {タイトル}

> **種別**: インシデント / 方法論 / author フィードバック / テンプレート
> **日付**: YYYY-MM-DD
> **関連ファイル**: （修正したファイルや参照先）
> **深刻度**: 高（canon 破壊）/ 中（手戻り発生）/ 低（改善機会）

## 何が起きたか

（事実を3〜5行で。時系列順に。何を書いて / 何が間違っていて / 誰がどう指摘したか）

## なぜ起きたか（根本原因）

（「うっかり」で終わらせない。構造的な原因を掘る）
- 確認不足なら: 何を確認すべきだったのに、なぜ確認しなかったか
- 思い込みなら: どんな前提を疑わなかったか
- 知識不足なら: どのファイルを読んでいれば防げたか
- 設計の穴なら: どの仕組みが欠けていたから起きたか

## どうやるべきだったか（正しい手順）

（同じ状況に直面した場合の正しいアクションを、手順レベルで書く）
1. まず○○を確認する
2. 次に○○と照合する
3. 矛盾があれば○○に戻す

## 実際にやった対処

（今回どう修正したか。変更したファイルと内容）

## 根本対応（再発防止策）

（同種の問題を仕組みで防ぐには何が必要か）
- rule / preflight に追加すべき項目はあるか
- ワークフローのどこにチェックポイントを入れるべきか
- 他の packet / 他の作品にも横展開すべきか

## 横展開チェック

（この問題と同じ構造を持つ既存の設定・ファイルを列挙。すでに別の場所で同じミスが埋まっていないか確認する）
```

## キャプチャしないもの

- 単なる作業ログ（「ep14 の scene card を書いた」）
- 既に learning/ に同趣旨のログがあるもの（重複禁止。既存ログを更新する）
- 一般論・教科書的な知識（この作品固有の学びだけ残す）

## 昇格パス

learning/ のログが繰り返し参照される場合:

- 執筆ルールに昇格 → `.claude/rules/` または `bible/rules.md` へ canon patch proposal を出す
- preflight に昇格 → `.claude/rules/drafter-preflight.md` に追記
- ワークフローに昇格 → `CLAUDE.md` の制作フローに追記

昇格は author の承認を得てから行う。learning/ の元ログは削除せず、昇格先を追記する。
